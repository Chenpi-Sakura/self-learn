## **0. 编写目的与读法**

本文档是**过渡性实施计划**，用于回答两个问题：

1. **目前 frontend 做到哪里了**（原名 `demo-serif/`，Stage 4 T1 重命名）？对照 v3 需求与详细设计，逐项核对完成度。
2. **接下来要做什么、按什么顺序做**？给出分阶段的实施路线图。

**读法建议**：

- § 1 是现状快照（一张表看清完成度）。
- § 2 是每个未完成项的具体差距分析（"差什么、为什么、怎么补"）。
- § 3 是**实施路线图**，按阶段切分，每阶段给出目标、产出、验收点。
- § 4 是**里程碑甘特与依赖图**，把阶段落到时间维度上。
- § 5 是**风险清单与应对**，避免实施时被技术债绊倒。

**本文档不重复** v3 需求与详细设计的内容，只标注"哪里有、对应文档节号"。凡涉及具体技术决策的，统一回引详细设计文档的节号，方便查证。

## **1. 完成度快照**

### 1.1 总体评估

`frontend/`（Stage 4 T1 重命名自 `demo-serif/`）是前端的**桌面交互骨架**——多窗口、布局切换、藏宝图可视化、画像雷达图、AI 对话悬浮窗的基础形态已经搭好（基于 Phase 1、Phase 1.5 的两个 commit）。**与 v3 详细设计文档相比，整体完成度约 20%~25%**，集中在"前端骨架与视觉规范"；后端、智能体、数据层、API、消息总线、资源生成等核心能力**全部未实现**。

### 2.3 demo-serif 与设计文档的不一致点（决策点）

这些是设计文档写得不明确或 demo-serif 实现走了岔路，需要决策后再继续：

| 不一致点 | 设计文档 | demo-serif 现状 | 建议 |
| --- | --- | --- | --- |
| **AppId 命名** | `treasure_map / chat / document / exercise / code_editor / notebook / mind_map / resource_library / dashboard / settings`（10 个） | `treasure_map / today / profile / calendar / doc / exercise / note / ...` | 把 `today / profile / calendar` 视为"demo 占位"，新增真实 `dashboard / settings` 后，把 `today` 替换成真实 `task_list`，`profile` 替换成真实 `profile`，`calendar` 作为 `dashboard` 的子视图或下掉 |
| **WindowState 字段** | `id / appId / title / position / size / zIndex / minimized / maximized / contentState / metadata / pinLevel`（11 字段） | `id / appId / x / y / w / h / z / minimized / maximized / pinLevel / _prev`（11 字段，命名不同） | 把 `x/y/w/h` 改为 `position/size` 对象；增加 `contentState / metadata` 字段；`_prev` 字段名加前缀保留 |

## **3. 实施路线图**

按依赖关系与可并行度切成 **5 个阶段**，每个阶段给出**目标、产出、验收点**。路线图的设计原则：

1. **垂直切片**：每个阶段都能交付一个"端到端可用"的最小闭环，避免半成品堆积。
2. **可演示**：每个阶段结束都能录一段 demo 给评委看。
3. **前后端分离推进**：前端骨架工作可以独立做（Stage 1/2/4/5），后端基础（Stage 3）单独推进，Stage 4 起交叉。
4. **文档先行**：每个阶段产出前，先在 `docs/` 下补完/调整设计文档（特别是 v3.5 之后的 6 项设计缺陷）。

## **5. 风险清单**

### 5.1 技术风险

| 风险 | 概率 | 影响 | 应对 |
| --- | --- | --- | --- |
| **LLM 生成质量不稳定** | 高 | 关卡内容可能不可用 | 第一版允许粗糙；评审 Agent 把好质量关；准备知识库种子数据兜底 |
| **评审 Agent 误杀率高** | 中 | 大量内容被拦截，正常流程跑不通 | 规则过滤先行；RAG 用低阈值；提供"强制通过"开发者选项 |
| **Pyodide 性能不足** | 中 | 代码窗口跑不动大模型 demo | 准备后端 Jupyter 方案作为 fallback |
| **WebSocket 流式断连** | 中 | 资源生成中断 | 设计文档已规定 Redis Stream 暂存 + 重连批量重放 |
| **讯飞 API 鉴权 / 限流** | 中 | 讯飞流程阻塞 | Week 11 Day 1 落实凭据；本地降级方案完整 |
| **知识库数据量不足** | 高 | 藏宝图生成出来只有 5 个节点 | S3 第一周确定数据来源；先人工写 30 个知识点 + LLM 校对 50 个 |
| **9 窗口开发工时溢出** | 高 | S4 拖延 | **M1 阶段就启动**：每个窗口先做最小可用版本；剩余时间迭代 |
| **持久化层（IndexedDB）兼容性** | 低 | Safari 隐私模式下失败 | 检测 + 降级到内存（带 Toast 提示） |

### 5.2 进度风险

| 风险 | 概率 | 影响 | 应对 |
| --- | --- | --- | --- |
| **关键技术决策拖到中后期** | 高 | 大规模返工 | § 4.3 列出的 8 个决策点必须在对应 Stage Day 1 敲定 |
| **后端 LLM 调通耗时超预期** | 高 | S3 阻塞 | 第一周就用 mock LLM 跑通流程，真实模型并行接入 |
| **评审发现 v3 设计缺陷** | 中 | 文档需要返工 | 预留每阶段 1-2 天文档维护时间 |
| **人员变动 / 任务交接** | 中 | 知识断层 | 每阶段产出文档必带"已知坑"清单 |

### 5.3 范围风险

| 风险 | 概率 | 影响 | 应对 |
| --- | --- | --- | --- |
| **加分项想全做** | 高 | 工期爆炸 | 加分项分级：必做（听力关卡 / ASR 转讲义）/ 可选（TTS 配音 / 视频生成）/ 不做（声音克隆 / 9 国同传） |
| **设计文档 v3 之后还有变更** | 中 | 实施与文档脱节 | 文档变更必须经过评审，并同步更新本文档 |
| **用户故事范围扩大** | 中 | 偏离 MVP | 严格按需求 § 1.3 的两个用户故事做基线，其余进 backlog |
